# 临时任务-触达式开发者文档

## 1. 文档范围
本文档仅说明“临时任务”中的“触达式任务（ACCOUNT）”开发口径，覆盖：
- 业务定义
- 权限与双中心模型
- 前后端关键链路
- 搜索添加 / 批量框选的组织范围口径
- 我的待办 / 看板 / 通知 / 发布者归属
- 已确认限制与注意事项

不包含：
- 日常任务
- 主播式任务（ANCHOR）
- 管理式任务（MANAGER）

---

## 2. 业务定义

### 2.1 触达式任务是什么
触达式任务对应临时任务模式：
- `temporaryMode = "ACCOUNT"`

其核心特征是：
- 任务面向“账号主体”投放
- 同一账号可映射多个身份
- 任一身份完成，即视为该账号主体完成

### 2.2 完成主体定义
触达式任务的完成主体不是“身份”，而是“账号”。

即：
- 一个账号可能存在多个身份
- 多个身份共享同一条任务记录
- 任一身份提交后，该账号主体任务即视为完成

这是当前系统确认成立的产品口径。

---

## 3. 双中心模型

触达式任务使用“双中心”模型：

### 3.1 第一中心：基地中心
任务发布与看板查询都需要明确基地归属。

用途：
- 任务归档
- 发布范围管理
- 看板基地维度筛选
- 草稿 / 发布 / 查询时的 `scopeOrgId`

### 3.2 第二中心：账号中心
任务的完成与分发主体以账号为核心。

用途：
- 任务记录主体使用 `USER:<userId>` 聚合
- 多身份共享任务可见性
- 任一身份完成即完成

### 3.3 与“发布者归属”的关系
触达式任务还额外遵循：
- 谁发布
- 谁管理
- 谁通知

即便任务归属到某个基地，也不代表同基地其他管理员天然可见或可管理。

## 4. 权限口径

### 4.1 发布角色
当前口径支持以下角色进入触达式临时任务发布：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`

### 4.2 看板角色
当前口径支持以下角色进入临时任务看板：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`

### 4.3 角色与基地中心关系
参考日常任务看板逻辑，当前临时任务收敛为：
- `DEV_ADMIN / HQ_ADMIN`：可显式选择基地
- `BASE_ADMIN / TEAM_ADMIN`：自动落所属基地，不提供跨基地切换

注意：
- `TEAM_ADMIN` 的稳定进入依赖“团队身份可正确映射到所属基地”
- 若组织树或身份 scope 数据异常，会导致“暂无可管理基地”

---

## 5. 组织树与基地映射要求

### 5.1 为什么 TEAM_ADMIN 容易出问题
`TEAM_ADMIN` 的身份中心是“团队”，但临时任务要求先建立“基地中心”。

因此系统需要从以下数据反推出所属基地：
- `currentIdentity.orgId`
- `currentIdentity.scopePath`
- `/orgs/tree` 返回的组织树父子关系

### 5.2 必须满足的数据条件
组织树中必须满足：
- TEAM 节点的 `parentId` 能追溯到 BASE
- TEAM 节点的 `path` 包含 BASE 路径
- `/orgs/tree` 返回结果中必须包含祖先 BASE 节点

### 5.3 已踩过的问题
曾出现以下问题导致 TEAM_ADMIN 无法进入临时任务发布 / 看板：
1. 前端页面只处理了 HQ/DEV 显式选基地，TEAM_ADMIN 无交互入口
2. `/orgs/tree` 对非开发管理员仅返回 scopePath 子树，不返回祖先 BASE
3. 后端补祖先节点时，路径拼接漏掉前导 `/`，导致祖先查询失效

最终结论：
- `TEAM_ADMIN` 无法进入的根因不是角色不够
- 而是“团队身份无法映射出所属基地中心”

## 6. 后端实现说明

### 6.1 模式定义
触达式任务在临时任务模式中对应：
- `ACCOUNT`

相关文件：
- `backend/src/modules/task/assignment/temporary-assignment.utils.ts`

### 6.2 主体聚合逻辑
在 `ACCOUNT` 模式下：
- 先收集显式账号 `targetUserIds`
- 再结合范围组织内账号
- 最终按 `userId` 聚合为 `USER:<userId>` 主体

核心特征：
- 同一个用户的多个身份会汇聚到一个主体
- 最终生成一条账号主体任务记录

### 6.3 任务记录创建
创建任务记录时：
- `subjectType = "USER"`
- `subjectKey = USER:<userId>`
- `subjectUserId = userId`

并将同账号下多个身份写入 `visibleIdentityLinks`

### 6.4 完成判定
提交任务时，更新的是同一条 `taskRecord`：
- 某个身份提交
- 即更新该账号主体 record 为完成

因此“任一身份完成 = 账号主体完成”是后端天然成立的。

## 8. 搜索添加与批量框选口径（2026-06 更新）

### 8.1 第三步的两种圈定方式
触达式任务第三步“组织圈定”当前分为两个入口：

1. `搜索添加`
   - 通过关键词搜索账号
   - 搜索结果来自账号搜索接口
   - 适合精确补人

2. `批量框选`
   - 通过组织树批量勾选组织
   - 发布时按账号聚合去重
   - 适合大范围圈定

两者的结果最终都会汇入：
- `selectedAccounts`
- `selectedOrgIds`

### 8.2 搜索接口口径
前端当前调用：
- `GET /accounts/search?keyword=...&scopeOrgId=...`

后端能力说明：
- `/accounts/search` 已支持通过 `scopeOrgId` 做组织范围过滤
- 后端会将 `scopeOrgId` 解析为组织 `path`
- 再通过账号可见身份的 `scopePath startsWith 组织path` 控制搜索范围

因此触达式“搜索不到团队管理账号”的问题，根因通常不是后端能力不足，
而是前端没有把正确的组织范围 ID 传进去。

### 8.3 本次修复前的问题
修复前，触达式 `搜索添加` 的 `scopeOrgId` 实际固定传的是：
- `managementOrgId`

这通常是当前页面的基地中心 ID。

导致：
- 搜索虽然在基地范围内执行
- 但不会自动跟随当前账号权限身份
- 团队管理员进入页面后，若希望只搜当前团队，就会产生“为什么团队管理搜索不到 / 搜不准”的体验问题

### 8.4 本次修复后的规则
当前实现已调整为：

#### 搜索范围默认值
搜索范围默认自动跟随“当前账号权限身份”定位：
- `BASE_ADMIN` → 默认基地
- `TEAM_ADMIN` → 默认团队
- `HALL_MANAGER`（若后续开放触达式）→ 默认厅

若当前身份组织不在可选组织范围中，则依次回退：
1. 当前身份组织
2. 当前身份父组织
3. 当前管理基地
4. 第一个可用组织

#### 搜索请求
搜索时继续复用原有接口能力：
- `scopeOrgId = 自动定位后的组织ID`

即：
- 不改后端搜索协议
- 不改返回结构
- 只修正前端 `scopeOrgId` 的来源

### 8.5 当前是否需要展示“搜索范围”选择器
从产品与交互角度，当前有两种可行策略：

#### 方案 A：保留选择器
优点：
- 用户可手动切换搜索范围
- 更灵活

缺点：
- 页面会多一个理解成本
- 当系统已能自动定位权限组织时，选择器价值下降

#### 方案 B：不展示选择器，仅自动定位
优点：
- 页面更简洁
- 更符合“账号权限在哪，就搜哪”的直觉
- 技术风险最低

缺点：
- 用户失去手动切换范围的能力

当前建议：
- 若业务不要求手动切换搜索范围，推荐采用 **方案 B：自动定位 + 不展示**
- 内部仍保留 `searchScopeOrgId` 状态即可，后续若需要恢复 UI 选择器，改动也最小

### 8.6 为什么这是技术风险最小的修法
因为本次修复：
- 不修改 `/accounts/search` 协议
- 不修改权限校验逻辑
- 不修改数据库查询模型
- 不修改发布/保存草稿逻辑
- 只修改前端 `scopeOrgId` 的传值来源

这是当前影响面最小、最容易回归验证的实现方式。

## 9. 我的待办口径

### 9.1 发布者展示
触达式任务在“我的待办”中应展示：
- `发布者账号：xxx`

说明：
- 当前展示为账号标签，不再显示手机号
- 该信息来源于后端补充的 `assignment.publisher`

### 9.2 表单说明展示
触达式任务在“我的待办”卡片中会展示：
- `record.assignment.template.description`

当前规则：
- 紧凑卡片：最多 2 行
- 非紧凑卡片：最多 3 行

### 9.3 完成主体提示
触达式任务在待办卡片中应明确提示：
- 完成主体：当前账号（同账号任一身份完成即视为完成）

## 10. 看板归属与发布者归属

### 9.1 基地归属
临时任务看板列表会使用基地归属条件查询：
- `createdByOrgId = scopeOrgId`

这表示任务在基地维度可归档。

### 9.2 发布者归属
但触达式任务严格执行：
- 谁发布
- 谁管理
- 谁通知

因此在真正的详情 / 进度 / 通知链路上，还要额外校验：
- `createdByIdentityId`
- 或 `createdBy`

### 9.3 结果
这意味着：
- 同基地管理员不一定能看到彼此发布的触达式任务
- 基地只是归属中心，不是共享管理权限中心

---

## 11. 通知口径

触达式任务的通知遵循：
- 谁发布
- 谁管理
- 谁通知

当前含义：
- 不是谁能看板谁就能通知
- 而是只有任务发布者本人可以继续发起通知

这与“严格执行任务归发布者”一致。

---

## 12. 豁免口径

当前口径已统一为：
- 所有临时任务均不支持豁免

即以下模式均关闭豁免入口：
- 触达式任务（ACCOUNT）
- 主播式任务（ANCHOR）
- 管理式任务（MANAGER）

仅日常任务保留豁免。

前端位置：
- `frontend/src/pages/tasks/my-todos/TaskRecordCard.tsx`

## 13. 关键字段说明

### 13.1 任务归属相关
- `scopeOrgId`：当前操作使用的基地上下文
- `createdByOrgId`：任务归属基地
- `createdByIdentityId`：发布身份
- `createdBy`：发布用户

### 13.2 主体相关
- `temporaryMode`：临时任务模式，触达式为 `ACCOUNT`
- `subjectType`：触达式通常为 `USER`
- `subjectKey`：账号主体键，格式 `USER:<userId>`
- `subjectUserId`：账号主体 userId

### 13.3 可见性相关
- `visibleIdentityLinks`：同账号下可见身份集合
- `lastSubmittedByIdentityId`：最后提交身份
- `lastSubmittedByUserId`：最后提交账号

## 14. 已确认产品口径

### 14.1 已成立
1. 触达式任务完成主体以账号为核心
2. 同账号任一身份完成即视为完成
3. 我的待办需展示发布者账号
4. 触达式任务不支持豁免
5. 通知遵循“谁发布谁管理谁通知”
6. 看板 / 详情严格执行“任务归发布者”

### 14.2 需长期注意
1. TEAM_ADMIN 进入链路依赖基地映射稳定
2. `/orgs/tree` 必须返回祖先 BASE 节点
3. path 字段必须统一保留前导 `/`
4. 不要把“基地归属”误解为“同基地共享可见”

## 15. 排查建议

若再次出现“TEAM_ADMIN 无法进入临时任务发布 / 看板”，优先检查：

### 15.1 当前身份
- `roleCode`
- `orgId`
- `scopePath`
- `org.name`
- `org.parentId`

### 15.2 组织树
- TEAM 节点的 `path`
- TEAM 节点的 `parentId`
- BASE 节点是否存在于 `/orgs/tree` 返回结果中
- TEAM 是否能追溯到 BASE

### 15.3 页面状态
- `availableBaseOrgs`
- `resolvedIdentityBaseId`
- `selectedScopeOrgId`

## 17. 总结
触达式任务的核心不是“身份任务”，而是“账号任务”。

其开发实现应始终围绕四条主线保持一致：
1. 基地中心必须建立成功
2. 账号主体必须共享完成
3. 发布者归属必须严格收口
4. 搜索添加的组织范围必须与当前账号权限身份保持一致

只要这三条一致，触达式任务的发布、待办、看板、通知链路就能保持稳定。
