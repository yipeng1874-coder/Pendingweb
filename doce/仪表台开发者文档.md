# 仪表台（驾驶舱）开发者文档

> 最后更新：2026-06-24（v2）

---

## 1. 功能概述

仪表台（Cockpit）是本系统 PC 端的核心数据看板页面，路由为 `/tasks/cockpit`。  
面向有管理权限的用户（BASE_ADMIN 及以上），集中展示：

| 模块 | 说明 |
|---|---|
| 顶部 Banner | 显示用户使用系统天数 |
| 我的待办汇总 | 当前身份下各类型任务的完成进度（饼图 + 卡片） |
| 历史待办完成率 | 昨天 / 近3天 / 近7天 / 本月（可自定义日期）的投放完成率进度条 |
| 主播汇总 | 基地维度的主播线上/线下分布、入职时间分布及各运营分组统计 |

## 4. 主播汇总模块

### 4.1 数据库模型

本次新增模型 `AnchorDailySummary`，对应数据库表 `anchor_daily_summaries`：

```prisma
model AnchorDailySummary {
  id            String   @id @default(uuid())
  baseOrgId     String   @unique @map("base_org_id")   // 每个基地只保留一条最新数据
  baseOrgName   String   @map("base_org_name")
  uploadDate    String   @map("upload_date")            // 格式 YYYY-MM-DD
  uploadedBy    String   @map("uploaded_by")            // 上传者 userId
  uploaderName  String   @map("uploader_name")          // 上传者昵称（快照）
  totalCount    Int      @map("total_count")
  onlineCount   Int      @map("online_count")
  offlineCount  Int      @map("offline_count")
  within7Days   Int      @map("within_7_days")          // 7天内入职主播数
  within30Days  Int      @map("within_30_days")         // 30天内入职主播数
  operatorStats Json     @map("operator_stats")         // 运营分组统计 JSON 数组
  rawRowCount   Int      @map("raw_row_count")          // 原始 Excel 行数
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  @@map("anchor_daily_summaries")
}
```

**设计说明：**
- `baseOrgId` 加了 `@unique`，每个基地只保留一条记录，上传时用 `upsert`
- `operatorStats` 存为 JSON，类型为 `OperatorStat[]`，无需额外关联表

### 4.2 operatorStats 结构

```typescript
type OperatorStat = {
  name: string;        // 运营名
  totalCount: number;  // 合计主播数
  onlineCount: number; // 线上主播数
  offlineCount: number;// 线下主播数
  within7Days: number; // 7天内入职
  within30Days: number;// 30天内入职
};
```

### 4.3 后端接口

#### 上传：`POST /api/anchor-summary/upload`

- **权限**：`task:report:view` + 角色不得为 `TEAM_ADMIN` / `HALL_MANAGER`
- **参数**：`multipart/form-data`，字段名 `file`，支持 `.xlsx` / `.xls`
- **query**：`scopeOrgId`（HQ_ADMIN / DEV_ADMIN 必须传）
- **必要列**：`主播昵称`、`所属基地`、`所属运营`、`主播类型`
- **线上/线下判断**：读取 `主播类型` 字段，值为 `线上` 则计入线上，否则计入线下
- **入职日期列**（可选）：兼容以下列名之一：`入职日期` / `加入时间` / `入职时间` / `加入日期`
- **存储方式**：`upsert`，同一基地每次上传覆盖上一次数据

#### 查询：`GET /api/anchor-summary/latest`

- **权限**：`task:report:view`
- **query**：`scopeOrgId`（HQ_ADMIN / DEV_ADMIN 必须传）
- **返回**：最新一条 `AnchorDailySummary`，无数据时返回 `null`

### 4.4 前端组件 AnchorSummaryCard

位于 `frontend/src/pages/tasks/cockpit/AnchorSummaryCard.tsx`，核心结构：

```
┌─────────────────────────────────────────────────────────┐
│ 标题行（h-14固定高度）                                    │
│  [图标+标题+日期/上传者（两行）] [合计/线上/线下/7天/30天] [错误小字] [刷新][上传] │
├─────────────────────────────────────────────────────────┤
│ OperatorRow × N                                         │
│  运营名  线下xx% ████░░░░░░░ 线上xx%  合计x 线上x 线下x 7天x 30天x │
└─────────────────────────────────────────────────────────┘
```

**进度条说明（PK 风格）：**
- 左侧红色渐变 = 线下占比
- 右侧蓝色渐变 = 线上占比
- 两端显示 `线下 x%` / `线上 x%` 文字标注
- 使用 `flex-1` 撑满中间区域，与左侧"历史待办完成率"保持一致的单行布局

**StatItem 展示顺序：** 标签在前、数字在后，例如"合计 382 线上 233 线下 149"。

### 4.5 上传结果弹窗提示（Toast）

上传操作结束后，使用项目公共组件 `shared/components/Toast.tsx` 弹出居中提示：

| 场景 | Toast 类型 | 显示时长 | 内容示例 |
|---|---|---|---|
| 上传成功 | `success`（绿色） | 3 秒 | 上传成功，数据已更新 |
| 上传失败 | `error`（红色） | 5 秒 | 表格缺少必要列：所属运营、主播类型 |

**双重提示机制：**
- **Toast 弹窗**：醒目、居中、有遮罩，带进入/退出动画，自动关闭
- **标题行红字**：持久显示在刷新按钮左侧（`text-[11px] text-red-500`），不会自动消失，作为长期提醒

两者并存：Toast 负责即时告知，标题行小字负责持久可见。

### 4.6 文件存储说明

上传的 Excel 文件**不落盘**，使用 `multer.memoryStorage()` 内存存储，后端解析完统计数据后立即释放。  
数据库中仅保存统计结果，**不保存原始行数据**。每次上传覆盖（`upsert`）同一基地的上一条记录。

## 5. 历史待办完成率模块

数据来源：`reportApi.getDailyRangeStats(start, end, scopeOrgId)`  
对应后端：`GET /api/task/report/daily-range-stats`

支持四个时间段：
- **昨天**：`yesterday ~ yesterday`
- **近3天**：`3天前 ~ 昨天`
- **近7天**：`7天前 ~ 昨天`
- **本月**：`本月1日 ~ 昨天`（支持自定义日期范围覆盖）

完成率颜色规则：
- `≥ 95%` → 绿色（`#10b981`）
- `≥ 80%` → 橙色（`#f59e0b`）
- `< 80%` → 红色（`#ef4444`）

## 7. 生产环境部署时的数据库迁移说明

> ⚠️ 这是最重要的注意事项。本次新增了 `AnchorDailySummary` 数据库模型，生产环境必须执行迁移，否则后端启动后调用相关接口会报错。

### 7.1 背景

本项目使用 **Prisma ORM**，数据库为 **MySQL**。  
每次在 `backend/prisma/schema.prisma` 中新增或修改模型后，必须将变更同步到数据库。

Prisma 提供两种方式：

| 方式 | 命令 | 适用场景 |
|---|---|---|
| `prisma migrate dev` | 开发环境生成迁移文件并应用 | 本地开发 |
| `prisma migrate deploy` | 生产环境执行已有迁移文件 | **上线部署** |
| `prisma db push` | 直接同步 schema 到数据库（不生成迁移文件） | 快速调试（不推荐用于生产） |

### 7.2 标准上线流程（含数据库变更）

当 `schema.prisma` 有改动时，在公司电脑（生产环境）执行以下完整流程：

#### 第一步：备份数据库（必做）

```bash
# 执行项目自带的备份脚本
db-backup.bat
```

或手动备份：

```bash
mysqldump -u root -p pendingweb > backup_before_migrate_YYYYMMDD.sql
```

#### 第二步：拉取最新代码

```bash
cd C:\deploy\source
git pull
```

#### 第三步：安装依赖（如有变化）

```bash
npm install
```

#### 第四步：执行数据库迁移

进入 backend 目录，运行迁移命令：

```bash
cd backend
npx prisma migrate deploy
```

> `migrate deploy` 会查找 `backend/prisma/migrations/` 目录下所有尚未执行的迁移文件，按顺序应用到生产数据库。  
> **前提**：开发环境在推送代码前已使用 `prisma migrate dev` 生成了对应的迁移文件，且该文件已包含在 Git 提交中。

如果因为历史原因没有迁移文件，也可使用（风险较高，仅在确认安全时使用）：

```bash
npx prisma db push
```

#### 第五步：重新构建后端

```bash
cd ..
npm run build -w backend
```

#### 第六步：重启后端服务

```bash
pm2 restart anchor-todo-api
```

#### 第七步：验证

```bash
pm2 logs anchor-todo-api --lines 30
```

观察日志，确认无启动错误，Prisma Client 正常初始化。

### 7.3 本次新增的迁移内容（参考）

本次 schema 新增了以下内容，对应迁移文件应包含：

```sql
CREATE TABLE `anchor_daily_summaries` (
  `id`             VARCHAR(191) NOT NULL,
  `base_org_id`    VARCHAR(191) NOT NULL,
  `base_org_name`  VARCHAR(191) NOT NULL,
  `upload_date`    VARCHAR(191) NOT NULL,
  `uploaded_by`    VARCHAR(191) NOT NULL,
  `uploader_name`  VARCHAR(191) NOT NULL,
  `total_count`    INT NOT NULL,
  `online_count`   INT NOT NULL,
  `offline_count`  INT NOT NULL,
  `within_7_days`  INT NOT NULL,
  `within_30_days` INT NOT NULL,
  `operator_stats` JSON NOT NULL,
  `raw_row_count`  INT NOT NULL,
  `created_at`     DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  `updated_at`     DATETIME(3) NOT NULL,

  UNIQUE INDEX `anchor_daily_summaries_base_org_id_key`(`base_org_id`),
  PRIMARY KEY (`id`)
) DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 7.4 快速判断是否需要执行迁移

在每次 `git pull` 后，检查以下内容是否有变化：

```bash
git diff HEAD~1 HEAD -- backend/prisma/schema.prisma
```

或查看迁移目录是否有新文件：

```bash
ls backend/prisma/migrations/
```

若有新增迁移文件夹（格式如 `20260624_add_anchor_daily_summary`），则**必须执行 `prisma migrate deploy`**，否则后端启动后相关表不存在，接口会报 500 错误。

### 7.5 常见迁移错误处理

| 错误信息 | 原因 | 处理方式 |
|---|---|---|
| `Table 'xxx' doesn't exist` | 未执行迁移 | 执行 `prisma migrate deploy` |
| `The table 'xxx' already exists` | 重复迁移或手动建表冲突 | 检查 `_prisma_migrations` 表，标记已执行 |
| `P3009: migrate found failed migrations` | 上次迁移中途失败 | 手动修复后执行 `prisma migrate resolve` |
| `Prisma Client is not generated` | Client 未重新生成 | 执行 `npx prisma generate` 后重新 build |

## 8. 开发时本地新增模型的流程

1. 修改 `backend/prisma/schema.prisma`
2. 执行：
   ```bash
   cd backend
   npx prisma migrate dev --name add_your_model_name
   ```
3. 自动生成迁移文件到 `backend/prisma/migrations/`
4. 提交迁移文件到 Git：
   ```bash
   git add backend/prisma/
   git commit -m "feat: add AnchorDailySummary model"
   ```
5. 推送到 GitHub 后，生产环境执行第 7.2 节流程即可
