# 飞书通知开发者文档

## 1. 文档目标

本文档用于系统性说明当前项目中的"飞书任务通知"能力，重点覆盖：

- 日常任务飞书通知的接口、筛选口径与页面入口
- 日常任务**自动定时通知**的接口、调度逻辑与配置方式
- 临时任务飞书通知的接口、筛选口径与页面入口
- 前后端代码目录、关键函数、关键字段与数据流
- 数据库表与字段如何支撑通知计算
- 后续开发者与 AI 工具接手时的推荐阅读顺序与维护注意事项

本文档的目标不是介绍飞书登录绑定，而是专门说明：

> 任务模块里，系统如何计算"谁该被通知"、如何调用飞书发送、以及前端在哪里触发通知。

---

## 2. 功能概述

当前项目中的飞书通知能力分为两条主线：

1. **日常任务飞书通知**（含手动发送和自动定时发送）
2. **临时任务飞书通知**

两者都基于：
- 用户表中的飞书绑定信息
- 飞书企业配置表中的 `appId / appSecret`
- 任务记录表中的任务完成状态

但两者在人群计算方式上有明显差异：

### 2.1 日常任务通知

日常任务通知更接近"按当天任务记录催办"。

主要特点：
- 面向某个基地当天的日常任务
- 只通知当天仍有**必填子任务未完成**的人
- 已被免除的任务不纳入通知
- 同一人按当天口径只统计一次
- 发送文案统一为：
  - `前缀，日常任务共 x 项，您还有 x 项未完成。`
- 支持**手动发送**和**自动定时发送**两种模式

### 2.2 临时任务通知

临时任务通知更接近"按某条进行中的 assignment 催办"。

主要特点：
- 只允许对**进行中的临时任务 assignment**发通知
- 只通知 `pending / in_progress` 的主体
- 人群计算复用临时任务现有主体分组逻辑
- 支持 `ACCOUNT / ANCHOR / MANAGER` 三种模式
- 业务规则当前按"主体统计不合并"实现预览与统计
- 发送文案当前统一为：
  - `前缀，临时任务共 x 件，您还有 x 件没完成。`

### 5.2.4 自动调度触发接口（内部调用）
- 函数：`runDailyNotifyScheduleTick(now?)`
- 文件：`backend/src/modules/task/notify/notify.routes.ts`
- 说明：由调度器（cron 或外部定时触发）调用，不对外暴露 HTTP 路由。

#### 调度规则
- 仅在每小时前 5 分钟（`getBeijingMinute < 5`）内响应，超过则跳过

- 以 `slotKey = "YYYY-MM-DD HH"` 为去重标识，防止同一时段重复发送
- 发送成功后更新 `DailyNotifySchedule.lastTriggeredSlot`

## 6. 权限与调用限制

## 6.1 通用中间件
所有通知接口统一挂在：
- `authRequired`
- `identityRequired`
- `permissionRequired("task:report:view")`

也就是说：
- 必须已登录
- 必须具备当前身份
- 必须拥有查看任务报表权限

## 6.2 日常任务权限规则
函数：`canManageDailyNotify(roleCode?: string)`

当前允许角色：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

此外还会通过 `resolveBaseScopeOrg` 解析基地范围：
- HQ / DEV 可显式选基地
- 其他角色从当前身份组织向上回溯所在基地

## 6.3 临时任务权限规则
当前实现中临时任务通知除了角色校验外，还要求：
- 任务必须是 `TEMPORARY`
- 任务必须是 `active`
- 发送者必须满足任务拥有权校验

任务拥有权规则来自：
- `backend/src/modules/task/assignment/assignment.service.ts`
- 关键函数：`assertTemporaryOwnership`

当前口径：
- 如果 assignment 有 `createdByIdentityId`
  - 当前身份 `identityId` 必须与之相同
- 否则回退到 `createdBy` 与当前 `userId` 匹配

这意味着：
> 当前临时任务通知不是"看板里谁能看谁就能发"，而是"任务发起人 / 发起身份"才能发。

## 7. 日常任务通知的人群计算逻辑

## 7.1 核心函数
- `buildDailyNotifyAudience(taskDate, baseOrg)`
- 文件：`backend/src/modules/task/notify/notify.routes.ts`

## 7.2 计算步骤
1. 对日期格式做标准化（将 `/` 替换为 `-`，防止前端传入斜杠格式导致查询失败）
2. 调用 `reconcileDailyAssignments(baseOrg.path)` 对日常任务状态做对账
3. 查询当前基地相关的日常任务 assignment
4. 过滤条件包括：
   - `category = DAILY`
   - `targets.some.orgId = baseOrg.id`
   - `status in (scheduled, active, ended)`
   - `deletedAt = null`
   - `effectiveAt <= 当天补录截止`
   - `endedAt` 为空或晚于当天结束时间
5. 对每个 assignment 调用：
   - `listAssignmentAudienceMembers(prisma, assignment)`
6. 以 `subjectKey` 去重（与看板对齐：同一 subjectKey 仅绑定到第一个 assignment）
7. 查询对应 `TaskRecord`（key 格式：`assignmentId:subjectKey:YYYY-MM-DD`）
8. **跳过**已被免除的记录（`exemption.status = approved / pending`）
9. 实时计算必填子任务完成数（`itemRecords` 中 `status=done && isRequired != false`）
10. **跳过**所有必填子任务已完成的用户（`requiredDoneItems >= totalCount`）
11. 拼出通知行 `NotifyAudienceRow`，按 `userId` 做聚合累加

## 7.3 通知文案中"任务数"的口径说明

这是最容易产生疑问的字段，明确如下：

**通知文案**：`前缀，日常任务共 N 项，您还有 M 项未完成。`

| 字段 | 含义 | 来源 |
|------|------|------|
| **N（总项数）** | 该用户当天所有日常任务中**必填子任务**的总数 | `template.items.filter(item => item.isRequired !== false).length` |
| **M（未完成数）** | 其中尚未完成的必填子任务数 | `N - requiredDoneItems`（已完成的必填子任务数） |

特殊情况：
- 若某日常任务**没有任何必填子任务**（`totalCount = 0`），系统将整条任务视为 1 件，`incompleteCount = 1`，兜底判断使用数据库 `status` 字段（`pending / in_progress`）
- 若用户当天被分配了**多条日常任务**，系统会累加所有任务的 N 和 M（`aggregateMap` 按 `userId` 汇总）

**注意**：这里统计的**只有日常任务（`category=DAILY`）**，不包含临时任务、专项任务或其他类型。

## 7.4 发送分组逻辑

同一飞书配置桶内，日常任务通知**按"N+M"相同的用户分为一批**发送：
- Key：`${totalCount}_${incompleteCount}`
- 相同 N 和 M 的用户合并到一次飞书批量发送调用中
- 不同 N 或 M 的用户各自单独发送，保证每条消息中的数字准确

这意味着：如果基地内有 10 人都是"共 5 项未完成 3 项"，则只需一次飞书 API 调用；但如果这 10 人的 N/M 各不相同，则需要 10 次调用。

## 7.5 关键口径说明
### 为什么日常任务是一人一天只统计一次
因为当前实现里：
- `resolvedSubjectKeys` 的 key 是 `subjectKey`（通常格式为 `USER:{userId}`）
- 同一用户当天重复命中时只保留第一次出现的 assignment 参考（按 `effectiveAt desc` 排序取最新）

### 为什么日常任务不支持"按 assignment 逐条通知"
因为日常通知本质是按"基地 + 当天"来汇总催办，而不是按单条 assignment。

## 8. 临时任务通知的人群计算逻辑

## 8.1 核心函数
- `buildTemporarySubjectGroups(...)`
- 文件：`backend/src/modules/task/assignment/temporary-assignment.utils.ts`

临时任务通知必须复用这套现有主体分组逻辑，不能自行重写组织树筛选。

## 8.2 三种模式

### A. `ACCOUNT`
主体类型通常按 `USER` 聚合。

来源包括：
- 明确设置的 `targetUserIds`
- 目标组织范围 `targets.orgPathSnapshot` 下扫描到的账号

### B. `ANCHOR`
主体类型按 `USER` 聚合。

来源包括：
- 目标组织范围下指定角色身份
- 排除组织与排除主播逻辑都会生效

### C. `MANAGER`
主体类型按 `ORG` 聚合。

来源包括：
- `temporarySubjectOrgType` 指定的组织层级
- 每个主体组织再反查其 `visibleIdentities`

## 8.3 当前通知口径
临时任务当前实现规则如下：
- 只允许通知 `assignment.status = active`
- 只通知 `TaskRecord.status in (pending, in_progress)`
- 统计上按主体计算，不做业务层合并
- 预览里显示：
  - 主体总数 `total`
  - 涉及账号数 `distinctUserCount`
  - 未开始数 `pendingCount`
  - 进行中数 `inProgressCount`
  - 未绑定飞书数 `unboundCount`

## 8.4 一个重要实现细节
### 业务统计"不合并"
你们当前定下的产品口径是：
- 一个人命中多条主体，不合并

因此在统计和预览上：
- `total` 统计的是主体数，不是用户数

### 发送层目前仍按 openId 去重批量发
后端当前复用飞书批量发送接口：
- `sendFeishuBatchMessage(config, openIds, text)`

因此同一飞书配置桶内：
- 相同 `openId` 会去重后再发

这意味着：
- 预览与统计口径：按主体
- 实际同一桶的飞书调用：按 `openId` 去重

如果未来要实现：
> 同一用户命中 3 条主体，就真的收到 3 条独立飞书

则需要把临时任务通知改造为：
- 不使用批量接口聚合 openIds
- 改为逐主体逐条发送

这是后续维护中最值得注意的一个点。

## 9.1 获取企业 token

- `POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal`

依赖字段：
- `config.appId`
- `config.appSecret`

## 9.2 批量发送消息
函数：`sendFeishuBatchMessage(config, openIds, text)`

调用飞书接口：
- `POST https://open.feishu.cn/open-apis/message/v4/batch_send/`

发送格式：
- `msg_type = text`
- `open_ids = [...]`
- `content.text = 文本`

## 9.3 为什么按飞书配置分桶
因为不同用户可能绑定在不同飞书企业配置下：
- 每个 `feishuConfigId` 对应不同 `appId / appSecret`
- 同一个企业配置共用一个 `tenant_access_token`

因此后端会：
1. 按 `feishuConfigId` 对通知对象分桶
2. 每桶单独换取 token
3. 每桶单独调用一次（或多次）飞书批量发送

## 13. 当前维护注意事项

## 13.1 不要单独重写临时任务通知的人群算法
临时任务通知一定要复用：
- `buildTemporarySubjectGroups`

否则很容易丢失：
- `ACCOUNT / ANCHOR / MANAGER` 三模式差异
- 组织排除规则
- 主播排除规则
- 可见身份逻辑

## 13.2 日常与临时的通知粒度不同
- 日常任务：基地 + 日期
- 临时任务：单 assignment

不要把两者混成同一套总通知逻辑。

## 13.3 临时任务的"业务统计不合并"和"发送去重"并存
这是当前实现里最容易误解的一点：
- 预览上看，临时任务按主体统计
- 发送上看，飞书批量接口会按 `openId` 去重

后续修改文案、统计口径、消息数量时，必须先确认业务想改的是：
- 统计展示层
- 还是实际飞书发送次数

## 13.4 日常任务通知中"N 项"的来源是必填子任务数
通知文案里的 N 和 M 都来源于模板的**必填 item** 计算，不是所有子任务，也不是其他任务类型：
- `N = template.items.filter(item => item.isRequired !== false).length`
- `M = N - 已完成的必填 item 数`

若某条日常任务无必填子任务，则系统将其整体计为"1 件"处理。

## 13.5 被免除的任务不纳入通知
若用户的 `TaskRecord` 存在 `exemption.status = approved` 或 `pending`，该记录会被直接跳过，不会出现在通知对象列表中。

## 13.6 日常任务发送按 N+M 分组
同一基地、同一飞书企业桶内，日常通知不是所有人共用一条消息，而是按 `totalCount + incompleteCount` 相同的用户合并发送：
- 这样保证每人收到的消息中数字是准确的
- 极端情况下（每人数值不同）可能产生较多飞书 API 调用

## 13.7 自动定时通知的防重复机制
`runDailyNotifyScheduleTick` 使用 `lastTriggeredSlot`（格式 `"YYYY-MM-DD HH"`）作为幂等 key：
- 同一时槽内如果已经发过，`status = "SKIPPED_DUPLICATE_SLOT"`
- 调度器应保证在每小时前 5 分钟内至少触发一次（`getBeijingMinute < 5` 才处理）

## 13.8 弹窗输入前缀是当前统一交互
目前通知前缀的交互已统一为：
- 页面主区域不再长期保留输入框
- 用户点击通知按钮后，在弹窗中填写前缀再确认发送

若后续开发者再次在页面主区域增加输入框，应先确认是否会破坏当前交互一致性。

## 13.9 推荐后续可继续优化的方向
后续如需继续扩展，建议优先考虑：

1. 将日常 / 临时通知弹窗抽成共用组件
2. 将通知预览 / 发送返回类型迁移到 `frontend/src/types.ts`
3. 引入通知发送日志表，记录：
   - 发送时间
   - 发送人
   - assignmentId / taskDate
   - 目标人数
   - 成功 / 失败人数
4. 若产品要求"一人命中多条主体必须收到多条消息"，改造临时任务发送为逐条单发
5. 考虑将自动通知调度迁移为系统内置 cron，而非依赖外部触发

## 14. 面向 AI 工具的阅读提示

如果后续让 AI 继续维护飞书通知能力，建议在提示词里明确说明：

1. 当前项目存在两套通知：`daily-feishu` 与 `temporary-feishu`
2. 日常任务还有独立的自动定时通知能力：`daily-schedule`
3. 日常任务与临时任务的人群计算逻辑不同
4. 临时任务通知必须复用 `buildTemporarySubjectGroups`
5. 日常任务通知文案中的"N 项"指**必填子任务数**，不是所有子任务数，也不包括其他任务类型
6. 临时任务当前的产品规则是：
   - 只通知 active assignment
   - 只通知 pending / in_progress
   - 业务统计按主体不合并
   - 文案前缀由发送者决定
7. 日常任务发送前需按 `totalCount+incompleteCount` 分组，保证每人收到的消息数字准确
8. 当前弹窗交互为统一页面内弹层，不应回退为浏览器原生 `window.confirm`

这样 AI 在继续改功能时，更不容易误删规则或重写错误的人群计算路径。
