# 临时任务-主播开发者文档

## 1. 文档范围
本文档仅说明临时任务中的"主播式任务（ANCHOR）"开发口径，覆盖：
- 业务定义
- 受众生成
- 发布 / 预览 / 看板 / 待办
- 组织树与范围映射
- 权限边界与收窄机制
- 已知风险点

不包含：
- 触达式任务（ACCOUNT）
- 管理式任务（MANAGER）
- 日常任务

---

## 2. 业务定义

### 2.1 主播式任务是什么
主播式任务对应临时任务模式：
- `temporaryMode = "ANCHOR"`

其核心特征是：
- 任务面向主播身份投放
- 发布时不让用户手动逐个圈主播
- 由当前管理范围自动生成主播受众
- 受众最终按账号主体聚合，账号下多个身份共享完成态

### 2.2 完成主体
主播式任务当前也是按账号主体归并记录：
- 一个账号可能存在多个主播相关身份
- 同账号的多个身份共享同一条任务记录
- 任一身份完成，即视为该账号主体完成

---

## 3. 双中心模型中的位置

主播式任务仍然处于临时任务的双中心体系下：

### 3.1 基地中心
用于确定当前任务归属、看板范围与发布上下文。

### 3.2 账号/主播中心
用于生成主播受众、完成态记录与可见身份列表。

主播式与触达式的区别在于：
- 主播式的受众是"主播身份集合"
- 触达式的受众是"账号主体集合"

---

## 4. 发布链路

### 4.1 前端入口
主播式任务在临时任务发布页中配置。

相关页面：
- `frontend/src/pages/tasks/manage/TemporaryIssuePage.tsx`
- `frontend/src/pages/tasks/manage/components/TemporaryIssuePanel.tsx`

### 4.2 管理范围
主播式任务不会让用户手动圈定组织树节点，而是使用当前管理范围作为投放基线：
- `managementOrgId`
- `managementScopePath`

其中：
- `managementOrgId` 会作为草稿与发布的 scope 基线
- `managementScopePath` 会参与组织树与可见范围判断

### 4.3 草稿保存
主播式草稿保存时：
- `mode = "ANCHOR"`
- `orgIds = [selectedScopeOrgId]`（前端传入，通常为当前管理基地的 orgId）
- `excludedAnchorProfileIds` 可配置
- `excludedOrgIds` 可配置
- `scopeOrgId = managementOrgId`

草稿保存阶段会做两层 scope 校验（见第 5.4 节）。

### 4.4 发布
发布时会基于草稿计算最终受众，并生成任务记录。

受众查询统一经过 `buildTemporarySubjectGroups`，该函数同时服务于：
- 预览（`getTemporaryPublishPreview`）
- 发布（`generateTemporaryRecords`）
- 飞书通知（`buildTemporaryNotifyAudience`）
- 自动催办（`temporary-notify-schedule`）

## 5. 受众生成规则

### 5.1 受众来源
主播式受众来自当前管理范围下的主播身份。

后端核心逻辑位于：
- `backend/src/modules/task/assignment/temporary-assignment.utils.ts`
  - 函数：`buildTemporarySubjectGroups`

### 5.2 受众筛选——三层过滤

当前实现是三层有序过滤：

**层 1（粗粒度圈定）**
1. 取草稿目标组织 `targets`，读取 `orgPathSnapshot`
2. 查询 `userIdentity.scopePath startsWith orgPathSnapshot` 且 `roleCode = ANCHOR` 的 active 身份

**层 2（发起者权限收窄）**  
基于 `assignment.ownerScopePath`（发起者自身 scopePath，保存在 assignment 表），对查出的身份做二次过滤：
- 只保留 `identity.scopePath === ownerScopePath` 或 `identity.scopePath.startsWith(ownerScopePath + "/")`的身份
- `ownerScopePath` 为 null 时（超级管理员或历史数据）跳过此层，行为退化为旧逻辑

**层 3（手动排除项）**  
按 `excludedOrgIds` 和 `excludedAnchorProfileIds` 排除手动圈定的排除项。

最终对剩余身份按 `userId` 聚合为账号主体组。

### 5.3 关键代码语义
主播式身份查询主要依赖：
- `identity.scopePath startsWith targetPath`

这意味着主播式受众的准确性，强依赖：
- 身份 scopePath 是否正确
- 组织树 path 是否完整
- 管理范围 `managementOrgId` 是否正确
- 发起者 `ownerScopePath` 是否正确写入

### 5.4 草稿保存阶段的 scope 校验

在 `saveTemporaryDraft`（`assignment.service.ts`）中，保存草稿时会对 `orgIds` 做 scope 合法性校验：

1. **组织存在性校验**：传入的 `orgIds` 必须在数据库中存在且 status = active
2. **路径合法性校验**：每个 org.path 必须与 `currentScopePath` 存在祖孙链关系（即：`org.path` 是 `currentScopePath` 的祖先、后代或本身）
3. 校验失败时抛出 403，防止发起者传入权限范围之外的 orgId

> **注意**：此层校验仅防止恶意构造请求，不承担受众收窄职责。受众收窄由层 2（ownerScopePath 过滤）在查询阶段完成。

## 6. ownerScopePath 字段说明

| 字段 | 位置 | 含义 |
|---|---|---|
| `ownerScopePath` | `assignment` 表 | 发起者保存草稿时的身份 scopePath，来自 `req.identity.scopePath` |

**各角色的 ownerScopePath 值：**

| 角色 | ownerScopePath | 说明 |
|---|---|---|
| `DEV_ADMIN` | `null` | 超级管理员，不做收窄 |
| `HQ_ADMIN` | `null` | 总部管理员，不做收窄 |
| `BASE_ADMIN` | `/QGCM001/SH2605A` | 基地 path，收窄到本基地 |
| `TEAM_ADMIN` | `/QGCM001/SH2605A/SH2605AA1` | 团队 path，只收窄到本团队 |

**效果示例（TEAM_ADMIN 发 ANCHOR 任务）：**

```
ownerScopePath = /A/BASE/TEAM1
orgPathSnapshot = /A/BASE（前端传基地 ID）

层1 查出：TEAM1/HALL001、TEAM1/HALL002、TEAM2/HALL003 的所有主播
层2 收窄：只保留 scopePath.startsWith("/A/BASE/TEAM1/") 的身份
        → TEAM2/HALL003 主播被过滤
层3 排除：手动排除项
```

## 7. 任务记录生成

主播式任务最终生成任务记录时：
- 以 `USER:<userId>` 或组织主体生成记录
- 为每个主体生成一条 `taskRecord`
- 同一账号的多个身份会共享同一条记录

生成记录后，系统会写入：
- `visibleIdentityLinks`
- `lastSubmittedByIdentityId`
- `lastSubmittedByUserId`

## 8. 看板链路

### 8.1 看板层级
主播式看板按以下层级展示：
1. 团队节点
2. 厅节点
3. 厅下主播记录
4. 记录详情

### 8.2 接口路径
常见接口包括：
- 临时任务看板任务列表
- 团队节点汇总
- 厅节点汇总
- 厅下记录列表
- 记录详情

### 8.3 懒加载
看板右侧详情是懒加载的：
- 先选中 assignment
- 再加载 summary
- 再加载团队 / 厅 / 记录详情

首次记录加载量通常为：
- 10 条分页

## 9. 我的待办展示

主播式任务在"我的待办"里应展示：
- 任务标题
- 表单说明 `template.description`
- 发布者信息
- 当前完成主体说明

当前已确认：
- 临时任务在紧凑模式下会展示发布者
- 主播式任务不支持豁免

## 10. 豁免口径

当前临时任务统一不支持豁免，包括：
- 触达式任务（ACCOUNT）
- 主播式任务（ANCHOR）
- 管理式任务（MANAGER）

只保留日常任务豁免。

## 11. 主要风险点

### 11.1 管理范围偏移
如果 `managementOrgId` 解析错误，主播式任务会整批发偏。

### 11.2 身份 scopePath 不准确
主播式任务受众筛选依赖 `userIdentity.scopePath`，scopePath 不准会导致：
- 发多
- 发少
- 排除不准

### 11.3 组织树数据不完整
如果组织树 parentId / path 链不完整，团队-厅-主播钻取会失真。

### 11.4 全量查询分页
主播式记录列表存在"先全量查，再内存分页"的实现，数据量大时有性能风险。

### 11.5 ~~TEAM_ADMIN 越权查整个基地主播~~（已修复）
> **历史问题，已于 2026-06-23 修复。**
>
> 问题：TEAM_ADMIN 发 ANCHOR 任务时，前端传基地 orgId，后端用基地 path 做 `startsWith` 查询，
> 会查出整个基地所有团队的主播，而不只是 TEAM_ADMIN 自己团队的主播。
>
> 修复：在 `buildTemporarySubjectGroups` 中，层1 查出身份后，通过 `ownerScopePath`（发起者 scopePath）
> 做层2 收窄过滤，只保留在发起者权限范围内的受众。
> `ownerScopePath = null` 时跳过收窄，旧数据和超级管理员不受影响。

## 12. 已知适配规则

### 12.1 发布角色
当前支持：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`

### 12.2 基地选择
当前收敛逻辑为：
- `DEV_ADMIN / HQ_ADMIN`：可显式切换基地
- `BASE_ADMIN / TEAM_ADMIN`：自动落所属基地

| 角色 | 可查受众范围 |
|---|---|
| `DEV_ADMIN` | 全平台不限 |
| `HQ_ADMIN` | 全平台不限 |
| `BASE_ADMIN` | 仅本基地下所有主播 |
| `TEAM_ADMIN` | 仅本团队下所有主播 |

## 14. 建议排查顺序

如果主播式任务出现"发错人/漏人"，建议按以下顺序排查：
1. `managementOrgId` 是否正确
2. `/orgs/tree` 是否返回祖先基地与团队节点
3. `userIdentity.scopePath` 是否与组织树 path 一致
4. `assignment.ownerScopePath` 是否正确写入（是否与发起者的实际 scopePath 一致）
5. `excludedOrgIds / excludedAnchorProfileIds` 是否命中
6. 发布预览中的 `subjectCount` / `visibleIdentityCount` 是否与预期一致

---

## 15. 结论
主播式任务当前实现可用，准确性依赖：
- 管理范围（`managementOrgId`）
- 组织树路径（org.path）
- 主播身份 scopePath（`userIdentity.scopePath`）
- 发起者身份 scopePath（`ownerScopePath`，用于权限收窄）

如果这四者保持一致，主播式任务可稳定按角色权限边界投放；否则会存在发偏、漏发或排除失效的风险。
