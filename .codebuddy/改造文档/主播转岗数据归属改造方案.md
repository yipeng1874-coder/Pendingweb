# 主播转岗数据归属改造方案

> 文档版本：v2.1  
> 创建时间：2026-06-17  
> 更新时间：2026-06-17  
> 背景：解决主播跨厅转岗后，历史任务数据"随人走"导致的组织归属错乱问题，并完善审计追踪、账号安全隔离及主播身份生命周期彻底终结机制

---

## 一、问题背景

### 1.1 核心问题描述

当主播小明从 **A 厅** 转入 **B 厅** 后，系统现有逻辑会：

1. 将 `AnchorProfile.hallOrgId` 直接更新为 B 厅 ID
2. 将小明名下所有 `UserIdentity`（`roleCode=ANCHOR`）的 `orgId` 批量更新为 B 厅
3. 将小明历史所有 `TaskRecord` 的 `subjectOrgId` 批量更新为 B 厅

**后果：**
- A 厅管理员永久失去了小明在 A 厅时产生的历史任务数据
- B 厅凭空"继承"了小明在 A 厅的所有历史数据，违背真实业务归属

### 1.2 问题代码定位

**文件：** `backend/src/modules/anchor/service.ts` → `updateProfile()` 方法

```typescript
// 第 508 - 521 行（问题根源）
if (nextHallOrgId !== current.hallOrgId) {

  await tx.userIdentity.updateMany({
    where: { anchorProfileId: updated.id, roleCode: "ANCHOR" },
    data: { orgId: nextHallOrgId, scopePath: nextHallPath },
  });
  
  // 问题2：强制将所有历史任务的 subjectOrgId 改为新厅，历史数据被"洗白"
  await tx.taskRecord.updateMany({
    where: {
      OR: [
        ...(updated.boundUserId ? [{ subjectUserId: updated.boundUserId }, { userId: updated.boundUserId }] : []),
        ...(current.douyinUid ? [{ subjectKey: current.douyinUid }] : []),

}
```

## 二、设计原则

### 2.1 核心原则：业务事件不可篡改

> **"已经发生过的业务事件，其所属组织归属必须在发生时永久固化，不得因主播的身份变动而事后修改。"**

### 2.2 数据模型层次关系

```
User（账号，全局唯一）
  └── UserIdentity（身份，一个账号可有多个身份）
        ├── roleCode: ANCHOR | ADMIN | ...
        ├── orgId: 所属组织（身份生效时绑定）
        └── status: active | disabled（时效状态）

AnchorProfile（主播档案，抖音 UID 全局唯一）
  └── hallOrgId: 当前所属厅（档案层面的归属）

TaskRecord（任务记录）
  ├── identityId → UserIdentity（产生记录时的身份）
  ├── subjectOrgId → OrgUnit（产生记录时的组织 ID，快照）
  └── subjectKey: 记录的业务主键
```

---

## 三、改造方案：身份版本化（Identity Versioning）

### 3.1 方案核心思路

**废弃"修改"操作，改为"失效旧身份 + 新建新身份"。**

| 操作 | 原逻辑 | 新逻辑 |
| :--- | :--- | :--- |
| 主播转岗 | `UPDATE UserIdentity SET orgId = B厅` | `UPDATE UserIdentity SET status='disabled'` + `INSERT UserIdentity (orgId=B厅)` |
| 历史任务 | `UPDATE TaskRecord SET subjectOrgId = B厅` | **不操作**（历史数据保持原状） |
| 任务生成 | 不写入 `subjectOrgId` | 写入 `subjectOrgId = identity.orgId` |

### 3.2 转岗后数据状态示意

```
小明（User）
  ├── 身份-1（已失效）roleCode=ANCHOR | orgId=A厅 | status=disabled | expiredAt=转岗时间
  │     └── TaskRecord-1, TaskRecord-2, ...（所有在A厅时产生的历史记录）
  │
  ├── 身份-2（活跃） roleCode=ANCHOR | orgId=B厅 | status=active
  │     └── TaskRecord-3, TaskRecord-4, ...（转岗后在B厅产生的记录）
  │
  └── 身份-3（活跃） roleCode=ADMIN | orgId=某团队 | status=active（管理身份不受影响）
```

### 3.3 关于数据唯一性的说明

`UserIdentity` 表的唯一索引为：

```prisma
@@unique([userId, roleCode, orgId, anchorProfileId])
```

由于新旧身份的 `orgId` 不同（A厅 vs B厅），两条记录**可以在数据库中合法共存**，不会触发唯一性冲突。

### 步骤二：改造转岗逻辑（核心改动）

**文件：** `backend/src/modules/anchor/service.ts`

**改动位置：** `updateProfile()` 方法，第 508~522 行

**改动内容：**

```typescript
// ============ 改动前（有问题的代码）============
if (nextHallOrgId !== current.hallOrgId) {
  // 直接修改身份 orgId（会导致历史数据漂移）
  await tx.userIdentity.updateMany({
    where: { anchorProfileId: updated.id, roleCode: "ANCHOR" },
    data: { orgId: nextHallOrgId, scopePath: nextHallPath },
  });
  // 暴力改写历史任务归属（根本问题所在）
  await tx.taskRecord.updateMany({
    where: {
      OR: [
        ...(updated.boundUserId ? [{ subjectUserId: updated.boundUserId }, { userId: updated.boundUserId }] : []),
        ...(current.douyinUid ? [{ subjectKey: current.douyinUid }] : []),
      ],
    },
    data: { subjectOrgId: nextHallOrgId },
  });
}

// ============ 改动后（身份版本化方案）============
if (nextHallOrgId !== current.hallOrgId) {
  const transferTimestamp = new Date();
  const bkSuffix = `_bk_${transferTimestamp.toISOString().slice(0, 10).replace(/-/g, "")}`;

  // 1. 将旧主播档案的关键唯一字段加 bk 后缀，释放唯一键占位，并物理锁死该档案
  await tx.anchorProfile.update({
    where: { id: current.id },
    data: {
      nickname:   `${current.nickname}(bk)`,
      douyinNo:   current.douyinNo  ? `${current.douyinNo}${bkSuffix}`  : undefined,
      douyinUid:  current.douyinUid ? `${current.douyinUid}${bkSuffix}` : undefined,
      status:     "transferred",  // 专属终态，不可逆转
    },
  });

  // 2. 同步更新旧 TaskRecord 关联键（确保历史任务依然能链接到旧档案）
  if (current.douyinUid) {
    await tx.taskRecord.updateMany({
      where: { subjectKey: current.douyinUid },
      data:  { subjectKey: `${current.douyinUid}${bkSuffix}` },
    });
  }

  // 3. 失效当前主播身份
  await tx.userIdentity.updateMany({
    where: {
      anchorProfileId: updated.id,
      roleCode: "ANCHOR",
      status: "active",
    },
    data: {
      status: "disabled",
      expiredAt: transferTimestamp,
    },
  });

  // 4. 在新厅创建全新的主播身份
  if (updated.boundUserId) {
    await tx.userIdentity.create({
      data: {
        userId: updated.boundUserId,
        roleCode: "ANCHOR",
        orgId: nextHallOrgId,
        anchorProfileId: updated.id,
        scopePath: nextHallPath,
        status: "active",
      },
    });
  }

  // 5. 写入转岗审计日志
  await tx.anchorTransferLog.create({
    data: {
      anchorId:       anchor.id,
      fromOrgId:      current.hallOrgId,
      toOrgId:        nextHallOrgId,
      fromProfileId:  current.id,
      toProfileId:    updated.id,
      transferTime:   transferTimestamp,
      transferReason: transferReason ?? null,   // 由接口入参传入
      operatorId:     currentAdminId,
    },
  });

  // 6. ✅ 不再执行 TaskRecord.subjectOrgId 的批量更新
  //    历史数据永远归属于原组织，这是正确的行为
}
```

### 步骤三：存量历史数据修复脚本

对于在改造上线前已经产生的、`subjectOrgId` 为空的 `TaskRecord` 记录，需要执行一次性数据修复。

**修复逻辑：** 根据 `TaskRecord.identityId` 关联到 `UserIdentity`，取 `UserIdentity.orgId` 回填到 `TaskRecord.subjectOrgId`。

**脚本思路（Prisma 版本）：**

```typescript
// scripts/backfill-subject-org-id.ts
// 用途：对存量 subjectOrgId 为空的任务记录，根据身份关联补填组织ID
async function backfillSubjectOrgId() {
  const records = await prisma.taskRecord.findMany({
    where: {
      subjectOrgId: null,
      identityId: { not: null },
    },
    include: {
      // 注意：identity 关联可能需要根据实际 schema 关系调整
    },
    take: 500, // 分批处理，避免锁表
  });

  for (const record of records) {
    // 通过 identityId 找到身份，取出 orgId
    const identity = await prisma.userIdentity.findUnique({
      where: { id: record.identityId! },
      select: { orgId: true },
    });
    if (identity?.orgId) {
      await prisma.taskRecord.update({
        where: { id: record.id },
        data: { subjectOrgId: identity.orgId },
      });
    }
  }

  console.log(`已处理 ${records.length} 条记录`);
}
```

> ⚠️ **注意**：此脚本需在**步骤二**上线后、且数据库低峰期运行，建议先在测试环境验证。

## 五、审计追踪改造（新增）

### 5.1 数据库模型：新增转岗日志表

在 `schema.prisma` 中新增以下内容：

```prisma
// 1. 在 AnchorStatus 枚举中增加 transferred 终态
enum AnchorStatus {
  unbound
  bound
  inactive
  transferred   // 已转出。专属终态，不可逆转，不可被任何接口重新启用。
}

// 2. 在 AnchorProfile 模型中增加审计关联字段
model AnchorProfile {
  id              String    @id @default(cuid())
  // ... 其他现有字段 ...

  // 审计追踪字段
  isTransferred   Boolean   @default(false)    // 是否是通过转岗生成的身份
  prevProfileId   String?                      // 指向转岗前的旧档案 ID（链式追溯用）
  transferAt      DateTime?                    // 本次入职/转岗的确切时间

  // 关联日志
  transferLogs    AnchorTransferLog[] @relation("ProfileLogs")
}

// 3. 新增专门的转岗历史记录表
model AnchorTransferLog {
  id              String   @id @default(cuid())
  anchorId        String   // 关联的 Anchor ID

  // 变更轨迹
  fromOrgId       String   // 原所属厅 ID
  toOrgId         String   // 新所属厅 ID
  fromProfileId   String   // 原档案 ID（转出方）
  toProfileId     String   // 新档案 ID（转入方）

  // 业务详情
  transferTime    DateTime @default(now())    // 执行时间
  transferReason  String?  @db.Text           // 转岗原因/方案备注
  operatorId      String                      // 执行操作的管理员 ID

  // 关联
  profile         AnchorProfile @relation("ProfileLogs", fields: [toProfileId], references: [id])

  @@index([anchorId])
  @@index([transferTime])
}
```

### 5.2 审计追踪价值说明

| 能力 | 实现方式 |
| :--- | :--- |
| **链式溯源** | 通过 `prevProfileId` 可从当前活跃档案一路追溯到主播最初入职的档案 |
| **责任明确** | `AnchorTransferLog.operatorId` 精确记录谁在何时执行了操作 |
| **方案留痕** | `transferReason` 字段存储转岗协议说明或备注 |
| **时间轴展示** | 基于 `transferTime` 可在后台展示主播的组织归属时间轴 |
| **不影响性能** | 审计数据存于独立表，不会使 `AnchorProfile` 查询变重 |

### 5.3 后续可实现的审查功能

1. **主播轨迹时间轴**：在主播详情页展示「A厅（2026-01~2026-06）→ B厅（2026-06至今）」的可视化轨迹。
2. **转岗报表导出**：导出指定时间段内所有转岗记录，包含原厅、新厅、操作人、转岗原因。
3. **操作人追责**：任何转岗操作均可溯源至具体管理员账号。

## 六、bk 标记与账号安全隔离方案（新增）

### 6.1 转岗后的账号状态变更规则

当主播从 A 厅转入 B 厅时，旧账号（留在 A 厅的归档记录）按以下规则处理：

| 字段 | 变更规则 | 示例 |
| :--- | :--- | :--- |
| **昵称** | 追加 `(bk)` 后缀 | `张三` → `张三(bk)` |
| **抖音号** | 追加 `_bk_YYYYMMDD` 后缀 | `DY123` → `DY123_bk_20260617` |
| **抖音 UID** | 追加 `_bk_YYYYMMDD` 后缀（**核心**：释放唯一键占位） | `UID_789` → `UID_789_bk_20260617` |
| **状态** | 改为 `transferred`（专属终态，不可逆） | `active` → `transferred` |

> 日期后缀 `YYYYMMDD` 的作用：防止同一主播多次转岗时，多个 bk 账号之间也产生唯一键冲突。

### 6.2 三层安全保护机制

转岗后的 bk 账号受到三层保护，业务风险被降为零：

| 层级 | 手段 | 效果 |
| :--- | :--- | :--- |
| **第一层：视觉标记** | 昵称加 `(bk)` 后缀，管理列表增加"已转出"标签 | 管理员一眼识别，不会操作错人 |
| **第二层：逻辑锁** | 状态为 `transferred` 时，所有启用/编辑接口报错拒绝 | 接口级别拦截所有重新启用的尝试 |
| **第三层：数据致残** | 抖音 UID 加后缀后，与抖音采集到的实时 UID 永远不匹配 | 即使绕过前两层，该账号也无法采集到任何实时业务数据 |

### 6.3 Service 层逻辑锁实现

**文件：** `backend/src/modules/anchor/service.ts`

在 `toggleProfileStatus`（或类似的启用/停用方法）中增加防御校验：

```typescript
async toggleProfileStatus(profile: any, enable: boolean) {
  // 核心防御：transferred 是不可逆终态，任何接口都不能将其重新激活
  if (profile.status === 'transferred') {
    throw createAnchorBusinessError(
      "CANNOT_REACTIVATE_TRANSFERRED_ANCHOR",
      "该账号已完成转岗归档，属于历史审计数据，禁止重新启用或修改。如有疑问请联系系统管理员。"
    );
  }

  // 原有的启用/停用逻辑...
}
```

同样，在任何**修改 `AnchorProfile` 字段**的接口入口处，均应增加相同的状态前置校验：

```typescript
// 通用前置守卫，建议抽取为独立函数
function assertProfileNotTransferred(profile: AnchorProfile) {
  if (profile.status === 'transferred') {
    throw createAnchorBusinessError(
      "CANNOT_MODIFY_TRANSFERRED_ANCHOR",
      "已转岗归档的账号为只读历史数据，不允许任何字段修改。"
    );
  }
}
```

### 6.4 前端展示策略

1. **主播管理列表**：默认过滤 `status: 'transferred'` 的记录，仅在"历史档案"或"转岗记录"专属模块中展示。
2. **状态标签渲染**：对 `transferred` 状态的行，渲染为灰色"已转出"标签，不渲染任何操作按钮（启用、编辑、删除）。
3. **搜索提示**：当管理员搜索主播昵称时，若同时找到活跃账号和带 `(bk)` 的档案，系统在 bk 账号旁显示"此为已转出的历史档案"提示。

### 7.2 登录与身份切换的兼容性
